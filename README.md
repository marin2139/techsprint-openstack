# TechSprint OpenStack Deployment

Automated deployment of isolated testing environments for Moodle applications using OpenStack infrastructure.

## Prerequisites

- OpenStack 16.1 (Red Hat Enterprise OpenStack Platform)
- Terraform >= 1.0
- Ansible >= 2.9
- OpenStack CLI tools
- Bash

## Features

- **Multi-tenant isolation**: Each developer gets isolated VNet
- **High availability**: 2 Moodle instances per developer with load balancer
- **Secure bastion access**: Jump host for secure access to developer environments
- **DevOps lead control**: Centralized management VM for team lead
- **Automated provisioning**: CSV-based infrastructure generation
- **Storage integration**: Object and file storage per developer

## Architecture

```
┌─────────────────────────────────────────────┐
│         External Network                     │
│         (provider-storage)                   │
└────────┬──────────────────────────┬──────────┘
         │                          │
    [Bastion VM]          [Lead VM - devops_lead]
    (Public IP)           (SSH access to all)
         │                          │
         └──────────────┬───────────┘
                        │
        ┌───────────────┼───────────────┐
        │               │               │
   [Dev Network 1]  [Dev Network 2]  [Dev Network N]
   - Moodle-1       - Moodle-1       - Moodle-1
   - Moodle-2       - Moodle-2       - Moodle-2
   - LB              - LB              - LB
   - Storage         - Storage         - Storage
```

## Quick Start

### 1. Prepare CSV file

Create `techsprint-users.csv`:

```csv
ime;prezime;rola
ana;anic;devops_lead
luka;lukic;developer
marko;marinkovic;developer
```

### 2. Generate SSH keys

```bash
ssh-keygen -t rsa -N "" -f ssh_key
```

### 3. Run deployment

```bash
./deploy.sh techsprint-users.csv
```

### 4. Monitor progress

The script will:
1. Parse CSV file
2. Initialize Terraform
3. Create OpenStack infrastructure
4. Run Ansible configuration playbooks
5. Output connection details

## Accessing the Environment

### Bastion Host (Jump Host)

```bash
ssh -i ssh_key root@<bastion-floating-ip>
```

### Moodle Instances (via Bastion)

```bash
ssh -i ssh_key -J root@<bastion-ip> root@<moodle-internal-ip>
```

### Lead VM (Direct access)

DevOps lead can connect directly to bastion and manage all resources.

## OpenStack Resources Created

### Per Developer

- Virtual Network (10.100.x.0/24 range)
- 2x Moodle VMs (2 vCPU, 4GB RAM)
- 2x Data volumes (20GB each)
- Load Balancer
- Security Groups (with SSH restricted to bastion)
- Object Storage bucket
- File Storage share

### Shared Resources

- Bastion VM (Jump Host)
- DevOps Lead VM
- Management Network
- Router with NAT

## Security Features

- SSH access only via bastion
- Per-developer network isolation
- Security groups with least-privilege access
- Lead VM can manage all instances
- Floating IP only on bastion

## Customization

### Change instance sizes

Edit `terraform/main.tf`:

```hcl
flavor_name = "m1.large"  # Change this
```

Available flavors: m1.small, m1.medium, m1.large, m1.xlarge

### Change network ranges

Edit `terraform/main.tf` subnet CIDR blocks:

```hcl
cidr = "10.X.0.0/24"  # Adjust as needed
```

### Add more developers

Simply add rows to `techsprint-users.csv`:

```csv
ime;prezime;rola
ivan;horvat;developer
marija;milic;developer
```

## Troubleshooting

### Terraform errors

Check OpenStack credentials in `terraform/terraform.tfvars`:

```bash
cat terraform/terraform.tfvars
```

### SSH connection issues

Ensure SSH key is readable:

```bash
chmod 600 ssh_key
```

### Ansible failures

Check inventory:

```bash
cat ansible/inventory.ini
```

Verify network connectivity from bastion:

```bash
ssh -i ssh_key root@<bastion-ip>
ping <moodle-ip>
```

## Cleanup

To destroy all resources:

```bash
cd terraform
terraform destroy -auto-approve
```
